📖 CORE LESSON

Understanding Data Graphs

Required • 30 minutes

The Five Key Concepts

Data Graphs are the bridge between your unified data and personalized content. Let's understand how they work:

1

What is a Data Graph?

A Data Graph is a curated subset of Data Cloud fields that you make available to Content Builder for personalization.

🛒 Analogy: Think of Data Cloud as a warehouse with thousands of products. A Data Graph is the display shelf you put in your store - you only show the products (fields) customers (marketers) actually need.

Why not expose everything?

  • Simplicity: Marketers don't want 500 fields - just the 5-10 they use regularly
  • Performance: Each field in a Data Graph is "hydrated" at send time (real-time lookup)
  • Governance: Control which fields are available (don't expose SSN, sensitive data)
2

Data Graph Architecture

Here's how Data Graphs work in the send flow:

Step 1
Segment identifies
Sarah Johnson
(Unified Individual)
→
Step 2
Data Graph looks up
her loyalty data
(tier, points)
→
Step 3
Email merges
personalized content
"Hi Sarah, Silver member"

This happens at send time - data is fresh and up-to-date!

3

Field Selection Strategy

When creating a Data Graph, you'll select fields from related DMOs. For NTO Rewards, we want:

  • From Individual DMO: FirstName, LastName, Email (identity fields)
  • From Loyalty Program Member DMO: MemberTier, PointsBalance, EnrollmentDate, MemberStatus
  • From Loyalty Promotion Enrollment DMO: PromotionName, EnrollmentDate (for trail participation)
💡 Key Insight: Start small! Add 5-10 fields for your first Data Graph. You can always add more later. Too many fields makes Content Builder overwhelming and can impact performance.
4

Consumption & Performance Considerations

Every Data Graph lookup consumes API calls. Here's what to know:

  • Real-time lookup: At send time, MC Next calls Data Cloud for each recipient's data
  • Consumption: Each send = 1 API call per recipient (500K send = 500K API calls)
  • Rate limits: Data Cloud has throughput limits (varies by SKU)
  • Caching: Data Graph responses are cached for short periods (reduces duplicate lookups)
⚠️ Important: For high-volume sends (1M+ emails), coordinate with Data Cloud team to ensure you have sufficient API capacity. Pre-warming may be needed.
5

The Deletion Problem (and Default Values)

What happens if a field is missing or a related record is deleted?

Example: Sarah is in your segment, but her Loyalty Program Member record was deleted.

  • Without default: Email shows "Hi , you have points" (blank merge fields)
  • With default: Email shows "Hi Member, join our loyalty program today!" (fallback content)

Best Practice: Always set default values for Data Graph fields:

FirstName: "Valued Customer"
MemberTier: "Member"
PointsBalance: "0"
💡 Pro Tip: Test your emails with records that have missing data to see how defaults render!

  • PointsBalance__c (current points)
  • EnrollmentDate__c (when they joined)
  • MemberStatus__c (Active, Inactive)

Also add these Individual fields:

  • FirstName
  • LastName
  • Email

Set Default Values

For each field, you'll see a "Default Value" column. Set these defaults:

FirstName: Valued Member
LastName: (leave blank - will use FirstName fallback)

MemberTier__c: Member
PointsBalance__c: 0
EnrollmentDate__c: (leave blank)
MemberStatus__c: Active

💡 Why these defaults? If someone is in your email segment but their loyalty record is missing, they'll see friendly fallback content instead of blank fields. "Hi Valued Member" is better than "Hi ,"!

Deep Dive: Data Graph Optimization

Optional • 30 minutes

Multi-Object Data Graphs

You can traverse multiple DMO relationships in a single Data Graph:

Example: Full NTO Member Profile Graph

  • Unified Individual → FirstName, LastName, Email
  • → Loyalty Program Member → Tier, Points, Enrollment Date
  • → Loyalty Promotion Enrollment → Trail Name, Status, Completion %
  • → Loyalty Ledger (last transaction) → Last Points Earned, Transaction Date

Use Case: "Hi Sarah, you earned 250 points on May 10th and you're 40% done with the Spring Hiking Challenge!"

Relationship Paths & Performance

Each relationship traversal adds query complexity:

1-Hop (Fast)

Unified Individual → Loyalty Program Member

Response Time: 50-100ms

Use for: Core personalization fields

2-Hop (Moderate)

Unified Individual → Loyalty Member → Promotion Enrollment

Response Time: 100-300ms

Use for: Engagement-specific data

3+ Hops (Slow)

Multiple deep relationships

Response Time: 300ms+

Use sparingly: Consider Calculated Insights instead

Data Graph vs Calculated Insights

When to use Data Graph:

  • Simple field lookups (name, tier, points)
  • 1-2 relationship hops
  • Data needs to be real-time fresh

When to use Calculated Insight:

  • Complex aggregations (SUM, COUNT, AVG)
  • Historical data (last 90 days purchase total)
  • Doesn't need to be real-time (daily refresh is OK)

Example: "Total Points Earned Last 90 Days" - calculate once daily as a Calculated Insight, then include that insight field in your Data Graph.

Handling One-to-Many Relationships

Problem: One member might have multiple Loyalty Promotion Enrollments. Which one do you show?

Solutions:

  • Filter in Data Graph: "Most recent enrollment only"
  • Use Content Builder logic: IF statements to show different trails based on status
  • Create separate Data Graphs: One per promotion type

Performance Optimization Tips

  • Keep field count low: Each field adds to response payload size
  • Use indexed fields when possible: MemberTier, MemberStatus are indexed
  • Cache at send time: MC Next caches Data Graph responses during batch sends
  • Monitor consumption dashboard: Set alerts for response time >500ms or error rate >2%

🏗️ IMPLEMENTATION NOTES

For Implementation Teams

Optional • 15 minutes

Discovery: Start Small Strategy

When planning Data Graphs with customers, recommend starting with a single, simple graph:

Phase 1: Core Personalization

Name, email, tier, points (5-10 fields max)

Timeline: Week 1-2

Phase 2: Engagement Data

Add promotion enrollment, activity completion

Timeline: Week 3-4

Phase 3: Historical/Calculated

Lifetime value, purchase history, behavioral scores

Timeline: Month 2+

Data Graph Naming Conventions

[Brand/Program] - [Purpose] - [Version]

Examples:
NTO Rewards - Core Personalization - v1
NTO Rewards - Trail Engagement - v1
NTO Retail - Purchase History - v2
Global - Contact Info - v1

Why versioning? If you need to make breaking changes, create a new version while old campaigns still use v1.

Expansion Planning: When to Add Fields

Add fields when:

  • Marketing team has specific use case (new campaign type)
  • Field is used in 3+ email templates (reusable)
  • Data is available and reliable in Data Cloud

Don't add fields when:

  • It's a one-time campaign (use different approach)
  • Data quality is poor (>20% missing values)
  • Field is sensitive/PII without clear approval

Testing Strategy Before Production

  1. Preview with 10+ test records: Bronze, Silver, Gold members + edge cases
  2. Test missing data: Verify defaults render correctly
  3. Performance test: Send to 1K recipients, check consumption metrics
  4. Pilot campaign: Send to internal team first (50-100 people)
  5. Monitor first production send: Watch consumption dashboard for errors

Common Pitfalls & Solutions

⚠️

Pitfall: Merge fields showing "%%FieldName%%"

Cause: Data Graph not activated, or wrong field API name

Solution: Verify Data Graph status = Active, check exact field API name (case-sensitive!)

⚠️

Pitfall: High error rate in consumption metrics

Causes: Segment includes non-loyalty members, relationship path broken, API timeout

Solution: Filter segment to only include records with loyalty data, check DMO relationships

⚠️

Pitfall: Slow send times (throttling)

Cause: Hitting Data Cloud API rate limits

Solution: Schedule sends during off-peak hours, increase API capacity (contact Salesforce), optimize Data Graph

When to Escalate

  • Performance issues: If Data Graph response time >500ms, engage Data Cloud architect
  • High-volume sends: For 1M+ recipient sends, coordinate with Salesforce (pre-warming)
  • Complex relationships: If need 4+ DMO hops, consult architecture team for alternatives
  • Multi-org scenarios: If Data Cloud spans multiple orgs, engage Professional Services
