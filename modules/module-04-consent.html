Understanding Consent Management

Required • 20 minutes

The Five Key Concepts

Consent management is critical for compliance and customer trust. Let's understand how it works in MC Next:

1

Communication Subscription Consent DMO

This is the Data Cloud object that stores consent status for each channel:

  • Individual ID: Links to the person
  • Channel: Email, SMS, WhatsApp, etc.
  • Consent Status: OptIn, OptOut, NotSet
  • Consent Date: When preference was last updated
  • Consent Source: Where it came from (Preference Center, API, Import)
💡 Key Insight: Each person can have multiple consent records - one per channel. Sarah might opt-in to Email but opt-out of SMS.
2

Consent Cache Layer (90-Day TTL)

For performance, MC Next doesn't check Data Cloud for every send. Instead, it caches consent status:

preference
(OptOut)
→
Step 2
Writes to

updates
(within minutes)

TTL (Time to Live): Cache entries expire after 90 days and refresh from Data Cloud.

⚠️ Critical: If consent data in Data Cloud is deleted, the cache still holds it for up to 90 days. This is the "deletion problem" - see Advanced section for details.
3

Safe Operating Patterns

To stay compliant, follow these patterns:

  • Pattern 1: Segment Filtering - Exclude opted-out members in your segment criteria (don't rely on MC Next to filter at send time)
  • Pattern 2: Preference Center First - Always provide a preference center link in emails (legal requirement in many regions)
  • Pattern 3: Re-permission Campaigns - Periodically ask inactive members to reconfirm consent (recommended annually)
  • Pattern 4: Audit Trail - Log all consent changes with timestamp and source (required for GDPR)
4

Channel-Specific Consent

Different channels have different consent requirements:

  • Email: Most permissive - implied consent often acceptable (transactional emails don't require opt-in)
  • SMS: Strict opt-in required (TCPA in US) - must have explicit consent with clear terms
  • WhatsApp: Requires opt-in + phone number verification + 24-hour message window
💡 Pro Tip: For SMS and WhatsApp, use double opt-in (confirm via the channel itself) for extra protection.
5

Preference Center Architecture

The Preference Center is a web page where members manage their subscriptions:

  • Hosted by: MC Next (Salesforce-hosted URL)
  • Authenticated by: Email + subscriber key (no login required)
  • Customizable: Branding, fields, layout
  • Writes to: Communication Subscription Consent DMO

Preference Center Link: Every marketing email should include a link like:

Update your preferences: %%preference_center_url%%

Deep Dive: Consent Cache & Bi-Directional Sync

Optional • 30 minutes

The Consent Cache vs DMO Priority Problem

Understanding the two-tiered consent system is critical for compliance:

Tier 1: Data Cloud (DMO) - Source of Truth

  • Communication Subscription Consent DMO stores all consent records
  • Written to by: Preference Center, API imports, data streams
  • Query latency: 100-500ms (real-time lookup)

Tier 2: MC Next Cache - Performance Layer

  • In-memory cache of consent status for fast sends
  • Refreshes from DMO every few minutes (not real-time)
  • TTL: 90 days (entries expire and are re-fetched)
  • Query latency: <10ms (ultra-fast)

The Problem Scenario:

  1. Member opts out via preference center (writes to DMO)
  2. Their consent record in Data Cloud is accidentally deleted (data cleanup script)
  3. MC Next cache still has "OptOut" status (from before deletion)
  4. Cache expires after 90 days, tries to refresh from DMO
  5. DMO has no record → defaults to "NotSet" → member might get emails again!
⚠️ Critical: Never delete consent records from Data Cloud. If you need to "reset" consent, write a new record with ConsentStatus = "NotSet", don't delete the record.

Bi-Directional Sync Roadmap

Currently (2026), consent sync is:

  • Preference Center → Data Cloud: Real-time (works great)
  • Data Cloud → MC Next Cache: Near real-time (few minutes)
  • External System → Data Cloud: Via API (your responsibility)

Roadmap (check current release notes):

  • Bi-directional sync with CRM (Contact.HasOptedOutOfEmail syncs to/from Consent DMO)
  • Real-time consent API webhooks (notify external systems of opt-outs)
  • Consent analytics dashboard (track opt-in/out trends)

Supported vs Unsupported Consent Methods

Supported (Safe) Methods:

  • ✓ Preference Center updates
  • ✓ Data Cloud API writes to Consent DMO
  • ✓ Data Streams importing consent from source systems
  • ✓ Journey Builder "Update Consent" activity

Unsupported (Risky) Methods:

  • ✗ Directly updating CRM fields (won't sync to Consent DMO automatically)
  • ✗ Deleting consent records (breaks cache refresh)
  • ✗ Manually editing Consent DMO via SQL (no audit trail)

Drift Risk: When CRM and Data Cloud Disagree

Common scenario:

  • Sales team updates Contact.HasOptedOutOfEmail in CRM
  • But this doesn't automatically sync to Consent DMO (unless you build custom integration)
  • Now CRM says "OptOut" but Data Cloud says "OptIn" → member gets email → complaint!

Solution Options:

  1. Single Source of Truth: Use only Consent DMO (disable CRM opt-out fields)
  2. Custom Sync: Build Flow/Apex to sync CRM → Consent DMO on field change
  3. Preference Center Only: Direct all opt-out links to Preference Center (don't allow CRM edits)

🏗️ IMPLEMENTATION NOTES

For Implementation Teams

Optional • 15 minutes

Discovery Questions for Consent Strategy

Q: Where do you currently store consent/opt-out data?

Why it matters: Determines migration strategy to Consent DMO.

Q: Do you have existing unsubscribe/preference pages?

Why it matters: May need to redirect old URLs to new Preference Center.

Q: What regions do you operate in? (GDPR, CCPA, CASL, etc.)

Why it matters: Different compliance requirements (opt-in vs opt-out default).

Q: Do you have transactional emails that should bypass consent?

Why it matters: Order confirmations, password resets shouldn't check marketing consent.

Supported vs Unsupported Consent Methods

Supported (Recommended):

  • Preference Center (built-in, audit trail, compliant)
  • Data Cloud API (for custom integrations)
  • Journey Builder "Update Consent" activity

Unsupported (Avoid):

  • Direct CRM field updates (no sync to Consent DMO)
  • Email unsubscribe links to custom pages (loses audit trail)
  • Manual imports without ConsentSource/ConsentDate (compliance risk)

Migration Strategy for Existing Consent Data

  1. Audit current state: Where is consent stored now? (CRM fields, legacy ESP, CSV files?)
  2. Export consent records: Get historical opt-in/out data with timestamps
  3. Transform data: Map to Consent DMO format (IndividualId, Channel, ConsentStatus, ConsentDate, ConsentSource)
  4. Import via Data Stream: Load historical consent into Data Cloud
  5. Validate: Query Consent DMO to confirm all records imported
  6. Cutover: Redirect all new consent changes to Preference Center

Common Pitfalls & Solutions

⚠️

Pitfall: Member opted out but still getting emails

Causes: Cache not refreshed, segment doesn't filter on consent, wrong subscription linked

Solution: Check cache refresh status, verify segment excludes opted-out members, confirm subscription mapping

⚠️

Pitfall: Preference center updates not persisting

Causes: IndividualId mismatch, Consent DMO not writable, duplicate records

Solution: Verify email maps to correct Unified Individual, check Data Cloud permissions

⚠️

Pitfall: CRM and Data Cloud consent out of sync

Causes: No bi-directional sync configured, sales team updating CRM fields

Solution: Build sync automation (Flow/Apex) or make Consent DMO the single source of truth

When to Escalate

  • Complex compliance requirements: Multi-region with different opt-in/out rules → engage legal team
  • Large consent data migration: 10M+ consent records → engage Data Cloud team for bulk import strategy
  • Custom consent workflows: Double opt-in, age verification, parental consent → consult with architecture team
