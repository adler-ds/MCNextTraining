Understanding Loyalty Integration

Required • 30 minutes

The Four Key Concepts

MC Next integrates natively with Salesforce Loyalty Management through Data Cloud. Let's understand the core building blocks:

1

Loyalty DMOs (Data Model Objects)

Loyalty data flows from Loyalty Management to Data Cloud as structured DMOs:

  • LoyaltyProgramMember: Core member record (tier, enrollment date, status)
  • LoyaltyMemberTier: Current tier assignment (Bronze, Silver, Gold) with effective dates
  • LoyaltyProgramPromotion: Active promotions members can enroll in (engagement trails, double points events)
  • LoyaltyMemberPromotion: Member enrollment in specific promotions
  • JournalEntry: Points transactions (accruals, redemptions, expirations)
💡 Key Insight: These DMOs are automatically mapped to your Data Graph in Module 1. That's why you can personalize with fields like {{LoyaltyMember.Tier}} and {{LoyaltyMember.PointsBalance}}.
2

Loyalty Event Triggers

Loyalty Management publishes real-time events to Data Cloud that can trigger MC Next journeys:

Available loyalty events:

Member Tier Changed: Fires when Sarah upgrades from Silver → Gold
Points Accrued: Fires when Marcus earns 500 points from a purchase
Points Redeemed: Fires when Elena redeems 2,000 points for a voucher
Promotion Joined: Fires when member enrolls in engagement trail
Promotion Completed: Fires when member completes trail requirements

These events enable real-time, contextual marketing messages.

3

Data Streams & Ingestion Patterns

Real-time streaming: Loyalty events publish within seconds of occurring

  • Use for: Tier upgrade celebrations, promotion joins, time-sensitive messages
  • Example: Sarah upgrades to Gold at 2:00pm → Celebration email arrives at 2:02pm

Batch ingestion: Loyalty snapshots sync hourly/daily

  • Use for: Points balance reports, tier status checks, bulk segmentation
  • Example: Nightly batch updates all members' points balances for reporting
💡 Best Practice: Use real-time events for member-facing communications (celebrate moments!). Use batch for analytics and bulk operations.
4

Transaction History

The JournalEntry DMO tracks every points transaction:

What's tracked:

  • Accruals: Points earned (from purchases, activities, bonuses)
  • Redemptions: Points spent (vouchers, products, experiences)
  • Expirations: Points that expired (typically 12 months after accrual)
  • Adjustments: Manual corrections by support team

Use cases for NTO:

  • Segment by activity: "High earners" (earned 5,000+ points in last 30 days)
  • Predict churn: Members who haven't earned points in 90 days
  • Expiration alerts: Members with points expiring soon

🧪 LAB 2

Build Points Expiration Reminder Campaign

Hands-on • 60 minutes

🎯 Your Mission

Create a campaign to remind members about points expiring in 30 days. Query loyalty transaction history to find members with expiring points, create a segment, and send automated reminders.

✅ Success Criteria:
  • Query JournalEntry DMO for points expiring in next 30 days
  • Create "Points Expiring Soon" segment
  • Build segment-triggered journey to send reminder
  • Personalize with expiration date and points amount

📋 Step-by-Step Instructions

1

Query for Expiring Points

Go to Data Cloud → Query → Create Query.

Name: Members_with_Expiring_Points

Build query:

  • FROM: Individual
  • JOIN: LoyaltyProgramMember on Individual.Id = LoyaltyProgramMember.ContactId
  • JOIN: JournalEntry on LoyaltyProgramMember.Id = JournalEntry.LoyaltyMemberId
  • WHERE: JournalEntry.ExpirationDate BETWEEN Today AND Today + 30 days
  • AND: JournalEntry.EntryType = "Accrual"
  • AND: JournalEntry.Balance > 0 (points haven't been redeemed yet)

SQL query finding members with points expiring soon

📸 mod7-lab2-step1-query.jpg

💡 Pro Tip: This query looks at JournalEntry records (transactions) to find accruals that expire soon. Much more accurate than just looking at current points balance!
2

Create Segment from Query

From the query results, click Create Segment.

Configure segment:

  • Segment Name: Points_Expiring_30_Days
  • Description: Members with points expiring in next 30 days
  • Refresh Schedule: Daily at 6am (check every day for new expirations)

Daily-refreshing segment of members with expiring points

📸 mod7-lab2-step2-segment.jpg

3

Build Reminder Journey

Go to Journeys → Create Journey → Segment-triggered.

Configure:

  • Journey Name: NTO_Points_Expiration_Reminder_v1
  • Segment: Points_Expiring_30_Days
  • Entry Limit: Once per member per 30 days (don't spam if they keep ignoring)

Add an Email interaction immediately after entry:

  • Email Name: Points Expiration Reminder
  • Subject: ⏰ Your NTO Rewards points are expiring soon!

Deep Dive: Multi-Program Management & Transaction History

Optional • 30 minutes

Multi-Program Considerations

If NTO runs multiple loyalty programs (e.g., Retail Rewards + Travel Rewards), you need to:

  • Segment by program: Add LoyaltyProgramMember.ProgramId = "NTO_Rewards" to all queries
  • Separate Data Graphs: Create NTO_Retail_Graph and NTO_Travel_Graph for different personalization contexts
  • Cross-program logic: Use decision splits to route members based on which program(s) they're enrolled in

Voucher Management

Many loyalty programs let members convert points to vouchers (discount codes). Integration points:

Voucher Issued Event

Trigger journey when member redeems points for voucher.

Email content: Voucher code, expiration date, redemption instructions.

Voucher Expiring Soon

Query for vouchers expiring in 7 days, send reminder.

Benefit: Increases voucher utilization, prevents "forgotten" discounts.

Voucher Redemption

Event fires when voucher is used at checkout.

Follow-up: Thank you email + "Redeem more points" upsell.

Transaction History Deep-Dive

Advanced segmentation using JournalEntry data:

  • High-value members: Members who earned 10,000+ points in last 90 days
  • Redemption risk: Members with high balance but no redemptions in 6 months (consider incentive to redeem)
  • Churn prediction: Members with no accruals in 90 days (send re-engagement campaign)
  • Tier acceleration: Members within 500 points of next tier (motivate with "almost there!" message)

Real-Time vs Batch Processing

Decision framework for loyalty campaigns:

Use Case Processing Type Why
Tier upgrade celebration Real-time (event) Celebrate the moment, high emotional impact
Points expiration reminder Batch (daily) Not time-sensitive within a day
Promotion enrollment Real-time (event) Confirm enrollment immediately
Monthly points summary Batch (monthly) Historical report, scheduled send

🏗️ IMPLEMENTATION NOTES

For Implementation Teams

Optional • 15 minutes

Discovery Questions for Loyalty Customers

When scoping loyalty integration, ask:

Q: How many loyalty programs do you run?

Why it matters: Multi-program requires separate Data Graphs and segmentation strategy.

Q: What are your most important loyalty milestones to celebrate?

Why it matters: Determines which events to prioritize (tier changes, promotion completions, point thresholds).

Q: What's your points expiration policy?

Why it matters: Drives expiration reminder campaign timing and frequency.

Q: Do members redeem for vouchers, products, or both?

Why it matters: Determines voucher management flows needed.

Performance Considerations for Large Member Bases

If loyalty program has 1M+ members:

  • Batch size limits: Daily segment refreshes should process max 100K members at a time (use batch windows)
  • Event throttling: High-volume events (Points Accrued) may need rate limiting to prevent journey overload
  • Query optimization: Add indexes to JournalEntry.ExpirationDate and LoyaltyMember.Tier for faster queries
  • Data retention: Archive old transaction history (2+ years) to keep queries performant

Testing Loyalty Journeys

Special considerations for testing loyalty flows:

Test Scenario How to Test What to Verify

Points expiration Create test JournalEntry with near-future expiration Member joins segment, gets reminder
Multi-tier logic Test with members in each tier (Bronze, Silver, Gold) Decision splits route correctly
Promotion enrollment Enroll test member in promotion Enrollment event triggers journey

Common Pitfalls & Solutions

⚠️

Pitfall: Event data doesn't include expected fields

Symptom: {{NewTier}} shows blank in email

Solution: Check event payload structure in Data Streams. Use {{LoyaltyMember.Tier}} instead of {{NewTier}} if field not in event.

⚠️

Pitfall: Points balance in email doesn't match loyalty system

Symptom: Member sees different balance in email vs portal

Solution: Data Cloud syncs every 15 minutes. Add disclaimer: "Balance as of {{SyncTimestamp}}"

⚠️

Pitfall: Expiration query includes already-redeemed points

Symptom: Members get expiration alerts for points they already used

Solution: Add filter JournalEntry.Balance > 0 to exclude redeemed transactions

When to Escalate

  • Custom loyalty events: If business needs events not in standard Loyalty Management (e.g., "Member Referred Friend"), engage Data Cloud developer to create custom event stream
  • Complex tier logic: If tier qualification requires custom calculation, engage Loyalty Management consultant
  • API integrations: If loyalty data comes from external system (not Salesforce Loyalty Management), engage integration architect
